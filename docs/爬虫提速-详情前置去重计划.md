# 爬虫提速计划：详情页请求前置去重（ccgp + gdgpo）

> 编写日期：2026-07-16
> 状态：**待评审**
> 关联代码：`scraper/base.py`、`scraper/ccgp/`、`scraper/gdgpo.py`、`app/models.py`

---

## Context（为什么做这个改动）

当前三个采集器结构同构：`_scrape_page` 在返回 lead 列表前就**逐条请求详情页**补充字段，入库去重（`save_leads`）发生在详情请求**之后**。后果是同一条公告被关键词 A 和关键词 B 都命中时，详情页会被请求两次——第二次入库才发现已存在被丢弃，详情请求（含限速 sleep）全白做。跨天采集（365 天窗口）也会重复请求昨天已抓过的详情页。

以 ccgp 为例，完整一次采集约 264 次列表请求 + 5000+ 次详情请求，每次请求前 `time.sleep(3~5秒)`，总耗时约 5.5 小时，绝大部分时间耗在串行 sleep 上。重复详情请求估计占总量 30%+。

**本轮目标**：把去重判断从「入库时」提前到「请求详情页之前」，已存在的公告直接跳过详情请求（保留列表页已有字段即可），消除重复工作。**不引入并发、不改搜索词、不改 save_leads、不动 eia**。改动是纯收益：不增加任何被封风险（反而减少请求量），不漏抓（前置判断语义与入库完全一致，不能判断时回退原行为）。

---

## 范围与约束（已确认）

- **去重 key：与入库同语义**——`bidding_number` 非空按 `bidding_number` 查 DB；为空则按 `project_name + buyer_name` 联合查。**不用 `source_url`** 作去重 key。
- **应用范围：ccgp + gdgpo**。eia 本轮不做（eia 所有 adapter 都不设 `bidding_number`，且 `buyer_name` 多在详情页才有，前置去重覆盖有限；且 eia 已有 `delay 1~2s` 比 ccgp 快得多）。
- **不改**：`save_leads`（base.py:321-420）现有去重逻辑、搜索词、限速 sleep、并发、eia、进度/暂停/停止控制机制。

### 核心不漏抓保证
前置去重判断「已存在」时，**必须与 `save_leads`（base.py:341-352）语义完全一致**。判断「不能确定」（缺 key）时，**必须回退为「请求详情页」**，绝不能跳过。`save_leads` 最终兜底去重不变，所以前置去重只会「提前跳过」，最坏情况是「没跳过、照常请求」（退化为原行为），**不会漏抓**。

## 实现方案

### 改动 1：BaseScraper 新增「批量预查去重」方法 + 进程内缓存

**文件**：`scraper/base.py`

**1a. run() 开头初始化进程内缓存**（base.py:606 `logger.info` 之后、`create_task` 之前/之后均可）
```python
self._seen_keys = set()  # 本次 run 内已处理过的 key，跨关键词共享
```
Key 构造（与入库语义对齐）：
- `bidding_number` 非空 → `('bid', bidding_number)`
- 否则 `project_name` 非空 → `('pn', project_name, buyer_name)`
- 否则 → `None`（无法去重，标记「必须请求详情」）

**1b. 新增方法 `_prefetch_existing_keys(self, leads)`**（放在 save_leads 附近，base.py:420 之后）
- 入参：一页的 lead 列表（列表页字段，尚未请求详情）
- 返回：`set`，本页中**已存在于 DB 或本进程缓存**的 key
- 逻辑：
  1. 从 leads 提取所有可去重 key（按 1a 规则），分两组：`bid_set`（bidding_number 集合）、`pn_pairs`（(project_name,buyer_name) 元组集合）
  2. 先扣掉 `self._seen_keys` 里已有的（进程内缓存命中，不查 DB）
  3. 剩余的批量查 DB：
     - `Lead.query.filter(Lead.bidding_number.in_(list(remaining_bids)))` → 取出存在的 bidding_number
     - 联合键：`tuple_(Lead.project_name, Lead.buyer_name).in_(pairs)` 多列 IN 一次查完
  4. 命中的 key 加入 `self._seen_keys` 并返回
- **线程安全**：当前 run() 单线程，`_seen_keys` 是实例属性，无需加锁。注释说明此假设。`progress.py` 的锁与本次无关。
- **异常兜底**：查询出错时 `logger.exception` 后**返回空 set**（即「都不命中」），回退为「全部请求详情」——绝不因去重查询失败而漏抓。

**关键实现细节（批量查询写法）**

项目用 SQLAlchemy 2.0.51（uv.lock 确认）+ SQLite（config.py:6），`tuple_().in_()` 多列 IN 完全支持。

```python
from sqlalchemy import tuple_
existing = set()
# 1) bidding_number 批量（单列 IN）
to_query_bids = [b for b in bid_set if ('bid', b) not in self._seen_keys]
if to_query_bids:
    rows = Lead.query.filter(Lead.bidding_number.in_(to_query_bids)) \
        .with_entities(Lead.bidding_number).all()
    for (b,) in rows:
        existing.add(('bid', b))
# 2) (project_name, buyer_name) 联合批量（多列 IN，一次查询）
to_query_pns = [p for p in pn_pairs if ('pn', p[0], p[1]) not in self._seen_keys]
if to_query_pns:
    rows = Lead.query.filter(
        tuple_(Lead.project_name, Lead.buyer_name).in_(to_query_pns)
    ).with_entities(Lead.project_name, Lead.buyer_name).all()
    for pn, bn in rows:
        existing.add(('pn', pn, bn))
```

语义一致性论证：save_leads（base.py:341-352）是「bidding_number 非空先查编号，空且 project_name 非空才查联合键」。前置按同样规则构造 key：有编号的 lead 的 key 是 `('bid', b)`，无编号但有 project_name 的 key 是 `('pn', pn, bn)`。库内一条 lead 若有编号，它只会被第 1 步命中；若库内一条 lead 无编号且 (pn,bn) 与前置 lead 相同，它会被第 2 步命中——这正是 save_leads 的 `elif project_name` 分支会跳过的情况。两条查询互不串味（编号 key 与联合 key 是不同前缀的 set 元素）。**结论：按上述两次批量查询即可，语义与 save_leads 完全一致，无需额外 `bidding_number.is_(None)` 过滤。**

若运行时发现某后端不支持 `tuple_.in_()`（理论上 SQLite+SA2.0 不会），回退为 `or_(*[and_(Lead.project_name==pn, Lead.buyer_name==bn) ...])`，仍是一次查询，不是 N 次单查。

**1c. 不改 save_leads**。前置去重只影响「是否请求详情」，save_leads 仍对每条 lead（含跳过详情的）做最终去重入库。跳过详情的 lead 入库时大概率被 save_leads 判定已存在（因为前置就是查它已存在才跳过的），new_count 不变。**进度统计（done_pages/collected）不受影响**：base.py:663 的 `len(leads_data)` 仍是列表项数，日志含义不变；collected=new_count 仍准确。

### 改动 2：ccgp search.py —— 请求详情前跳过已存在项

**文件**：`scraper/ccgp/search.py`（第69-77行）

原：
```python
detailed_leads = []
for lead in leads:
    detail_url = lead.get('source_url', '')
    if detail_url:
        detail_data = scraper._fetch_detail(detail_url)
        if detail_data:
            lead.update(detail_data)
    detailed_leads.append(lead)
return detailed_leads
```

改为：
```python
# 请求详情页前先批量判断已存在项，跳过详情请求（消除重复工作）
existing_keys = scraper._prefetch_existing_keys(leads)
detailed_leads = []
skipped = 0
for lead in leads:
    key = scraper._lead_dedup_key(lead)  # 提取本条 key（见改动 1a 规则）
    if key and key in existing_keys:
        skipped += 1
        detailed_leads.append(lead)  # 保留列表页字段，不请求详情
        continue
    detail_url = lead.get('source_url', '')
    if detail_url:
        detail_data = scraper._fetch_detail(detail_url)
        if detail_data:
            lead.update(detail_data)
    detailed_leads.append(lead)
if skipped:
    logger.info('[ccgp] 跳过 %d 条已存在项的详情请求', skipped)
return detailed_leads
```

**功能影响评估**：
- 原行为：每条 lead 都请求详情页。
- 新行为：key 在 DB/缓存中已存在的 lead，跳过详情请求，保留列表页字段直接入库（入库时被 save_leads 去重，new_count 不增）。
- 不漏抓：跳过的都是「已存在」项，本来就不会新增；key 为 None（缺字段）的走原逻辑请求详情。
- 回退条件：`_prefetch_existing_keys` 出错返回空 set → 全部请求详情（原行为）。

### 改动 3：ccgp channel.py —— 同样模式

**文件**：`scraper/ccgp/channel.py`（第99-106行）
同样在第96行 `if not leads: return []` 之后、详情循环之前插入 `existing_keys = scraper._prefetch_existing_keys(leads)`，循环内加 `if key and key in existing_keys: append+continue`。模式与改动 2 完全一致。
注意频道列表无 bidding_number，key 多为 ('pn', pn, bn)，buyer_name 常空 → key 可能为 None → 多数回退请求详情。这是预期的（频道列表前置去重收益有限但零风险）。

### 改动 4：gdgpo.py —— 三个子流程分别处理

**文件**：`scraper/gdgpo.py`

**4a. maincms `_scrape_page`（第217-224行）**：在 `for row in rows:` 循环前，先 `lead_skeletons = [self._row_to_lead(row) for row in rows]`，再 `existing_keys = self._prefetch_existing_keys(lead_skeletons)`，循环内按 key 跳过 `_fetch_detail`。保留原 `_check_pause_and_stop()`（第219行）在循环顶部。
注意：原代码是「先映射再请求详情」，需把映射提前成批量，再按 key 决定是否请求详情。

**4b. gpfa征集 `_scrape_gpfa_page`（第274行附近）**：同样模式，`_gpfa_row_to_lead` 已在请求前设 bidding_number=projectCode，前置去重可按 bidding_number 跳过详情。

**4c. gpfa二次竞价 `_scrape_gpfa_xjj_page`（第412-419行）：不改。** 列表行无 bidding_number、无 buyer_name（第522-527行只有 project_name + source_url）。前置去重只能构造 `('pn', project_name, '')` 这种 key，而库内对应 lead 的 bidding_number 是详情页才补的 `projectNumber`（第451行）——入库时 save_leads 按 bidding_number 去重，与前置的联合 key 语义不一致，存在「前置按 (pn,'') 误判已存在而跳过详情，但库内实际是另一条同标题不同编号的公告」的错跳漏抓风险。遵循「不能确定就不跳过」原则，**该子流程保持原状，不做前置去重**。计划明确：gpfaxjj / gpfaresult 子流程本轮无提速，但行为正确、零漏抓风险。

## 验证方法

### 端到端不漏抓验证
1. **改前基线**：对一个小关键词集合（如 ccgp 取 2 个词）跑一次完整采集，记录新增条数 N1 和详情请求次数（看日志 `请求第 x/y 次` 里详情 URL 的出现次数，或临时在 `_fetch_detail` 加计数日志）。
2. **改后对比**：同样关键词集合再跑一次（此时 DB 已有这批数据），预期：
   - 新增条数 N2 ≈ 0（因为已入库，前置去重 + save_leads 双重跳过）
   - 详情请求次数大幅下降（跳过已存在项）
3. **清库重跑**：清空 leads 表后跑改后版本，新增条数应与改前基线 N1 一致（证明没漏抓），详情请求次数应 ≥ 改前（因为全量抓，无跳过）。
4. **关键词重叠测试**：选两个高度重叠的词（如「管道」和「PE管」），改后跑——预期第二个词命中与第一个词重叠的公告时，详情请求被跳过（看日志 `跳过 N 条已存在项的详情请求`）。

### 日志增强（临时，验证后可保留）
在 `_prefetch_existing_keys` 返回前 `logger.info('[%s] 本页 %d 条，已存在跳过 %d 条详情请求', ...)`，便于对比改动前后详情请求量。

### 单元层面
- `_lead_dedup_key`：构造规则与 save_leads 一致（bid 非空用 bid；空且 pn 非空用 pn+bn；都空返回 None）。
- `_prefetch_existing_keys`：DB 异常返回空 set（回退、不漏抓）。

### 风险点复查清单
- [ ] 跳过的 lead 仍 append 到返回列表（保证 save_leads 能处理，不丢列表字段）
- [ ] key=None 的项必走原详情请求分支（不漏抓）
- [ ] `_prefetch_existing_keys` 异常 → 空 set（回退原行为）
- [ ] 不改 save_leads，入库兜底不变
- [ ] 进程内 `_seen_keys` 在 run() 开头初始化（避免跨次 run 污染）
- [ ] gdgpo 各子流程保留 `_check_pause_and_stop()`

## 预期收益

- **ccgp**：跨关键词 + 跨页 + 跨天重复详情请求被跳过，预计减少 30%+ 详情请求，整体耗时从 ~5.5h 下降（具体取决于重叠度，估算降至 3.5~4h）。
- **gdgpo maincms / gpfa征集**：列表页多数有 bidding_number，前置去重命中率高，跨天重复请求大幅减少。
- **不改并发、不改限速**：零封号风险增量，纯减少无效请求。
- **后续可叠加**：本轮只做前置去重；若仍嫌慢，可再单独评估「详情页并发请求」方案（需重构限速为全局速率控制，预计可再降至 ~1.5h，但有反爬风险，需独立决策）。
