### 第 0 步：环境准备

**目标**：让 pytest 可在 venv 中正常运行。

1. 安装 pytest：

2. 验证：
   ```bash
   uv run pytest --collect-only -q
   ```
3. 在 `pyproject.toml` 添加 `[tool.pytest.ini_options]`：
   ```toml
   [tool.pytest.ini_options]
   testpaths = ["tests"]
   markers = [
     "live: 真实网络抓取烟雾测试，需 LIVE_SMOKE_TESTS=1",
     "playwright_required: 需要安装 Playwright + Chromium",

   addopts = "-q --strict-markers"
   ```

**验证标准**：`uv run pytest --collect-only -q` 能成功收集 tests/ 下所有测试节点。

### 第 1 步：建立共享测试基础设施

**目标**：创建 conftest.py，提供统一的 live 测试门控和辅助函数。

创建 `tests/conftest.py`，内容：

```python
import os
import tempfile
import pytest

# ── live 烟雾测试门控 ──
# 默认跳过所有 @pytest.mark.live 测试
# 设置环境变量 LIVE_SMOKE_TESTS=1 才运行
live_skip = pytest.mark.skipif(
    os.environ.get('LIVE_SMOKE_TESTS') != '1',
    reason='设置 LIVE_SMOKE_TESTS=1 才运行真实网络烟雾测试',
)

# ── Playwright 可用性检测 ──
playwright_skip = pytest.mark.skipif(
    not _check_playwright(),  # 检测 playwright + chromium 是否可用
    reason='Playwright 或 Chromium 未安装',
)

# ── API Key 可用性检测 ──
amap_key_skip = pytest.mark.skipif(
    not os.environ.get('AMAP_API_KEY'),
    reason='未设置 AMAP_API_KEY 环境变量',
)
qcc_key_skip = pytest.mark.skipif(
    not os.environ.get('QCC_API_KEY'),
    reason='未设置 QCC_API_KEY 环境变量',
)

# ── 临时 Flask app fixture ──
# 用内存 SQLite 避免污染 instance/hongtu.db
@pytest.fixture
def live_app():
    from app import create_app
    app = create_app({
        'SQLALCHEMY_DATABASE_URI': 'sqlite://',  # 内存 DB
        'SCRAPE_SAVE_SNAPSHOT': False,
        'SCRAPE_DOWNLOAD_ATTACHMENTS': False,
    })
    with app.app_context():
        from app.extensions import db
        db.create_all()
        yield app

# ── 公共断言辅助 ──
def assert_valid_leads(leads, required_fields=None, source_type=''):
    """断言线索列表非空且关键字段非空。

    Args:
        leads: list[dict] 线索列表
        required_fields: 必须非空的字段名列表，默认 ['project_name', 'source_url']
        source_type: 数据源标识，用于日志
    """
    if required_fields is None:
        required_fields = ['project_name', 'source_url']
    assert leads is not None, f'[{source_type}] _scrape_page 返回 None（请求失败）'
    assert isinstance(leads, list), f'[{source_type}] 返回类型不是 list'
    assert len(leads) > 0, f'[{source_type}] 未抓取到任何线索'
    for i, lead in enumerate(leads[:3]):  # 只检查前 3 条
        for field in required_fields:
            value = lead.get(field, '')
            assert value, f'[{source_type}] 第{i+1}条线索 {field} 为空'
```

**验证标准**：`uv run pytest tests/conftest.py -q` 无报错；`LIVE_SMOKE_TESTS=1 uv run pytest -m live --collect-only -q` 能收集 live 测试节点。

### 第 2 步：编写烟雾测试文件

**目标**：创建 `tests/test_live_smoke.py`，每个渠道 1-2 个测试方法，真实请求目标站。

#### 统一测试契约

每个渠道的烟雾测试遵循相同契约：

1. 实例化采集器（`app=None` 或 `live_app` fixture）
2. 关闭限速（`delay_min=0, delay_max=0`）
3. 调用低层抓取方法（`_scrape_page(keyword, 1)` 或等价方法）
4. 断言返回非空 list + 关键字段非空

**关键设计**：只调用 `_scrape_page()` 等低层方法，不调用 `run()`。原因：
- `run()` 会真实入库（写 Lead/ScrapeTask 表）、触发完整采集流程
- `_scrape_page()` 只做请求+解析，返回 `list[dict]`，不入库
- 烟雾测试只需验证"能抓到 + 能解析"，入库逻辑已有单元测试覆盖

### 第 4 步：修复失败的渠道

**目标**：对第 3 步失败的渠道，定位问题并修复。

常见失败原因与修复方向：

| 失败原因 | 修复方向 |
|---------|---------|
| 目标网站接口变更 | 更新 API 路径/参数 |
| 反爬升级（WAF/Captcha） | 评估是否可绕过，或调整请求策略 |
| 解析选择器失效 | 更新 parser.py 的 CSS/XPath 选择器 |
| Cookie 获取流程失败 | 更新 playwright_utils 或 Cookie 刷新逻辑 |
| 网络超时 | 调整 timeout/retry 参数 |
| 目标站下线 | 标记为已放弃 |

修复后重新运行第 3 步验证。

## 三、风险与权衡

| 风险 | 说明 | 对策 |
|------|------|------|
| 网络依赖 | live 测试依赖目标站可访问 | 失败时保存错误信息供人工排查 |
| 反爬触发 | 真实请求可能触发反爬/限流 | 每渠道只请求 1-2 次，关闭限速减少等待 |
| Playwright 环境 | pipebiz/fdtz 需 Chromium | 无 Playwright 时优雅 skip |
| API Key 缺失 | poi/qcc 需付费 Key | 无 Key 时优雅 skip + 报告标注 |
| 测试污染 | `_scrape_page` 不入库但可能触发目标站缓存 | 仅请求第 1 页，最小化影响 |

## 五、后续维护建议

1. **定期重跑**：建议每周或每月重跑 live 烟雾测试，及时发现网站变更
2. **自动化**：后续可考虑加入 CI（GitHub Actions），但当前项目无 CI 基础设施
3. **响应快照**：对失败的渠道，保存真实响应快照到 `tests/fixtures/<source>/` 便于离线排查
4. **扩展到深度验证**：烟雾测试通过后，可对关键渠道（ccgp/ggzyjy）做字段完整性验证
